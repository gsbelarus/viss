# Task Specification: Node.js Video Processing and Analysis Pipeline

Build a **Node.js backend pipeline** that processes an already-downloaded short video file, analyzes its multimodal content, and stores the final structured result in a database collection.

The system environment already has `ffmpeg` installed.
Use OpenAI for:

* transcription
* frame-level multimodal analysis
* final video-level reasoning
* embeddings generation

## Goal

Given a local video file path that already exists on the backend, the system must:

1. inspect the video;
2. extract technical metadata;
3. extract audio;
4. transcribe speech;
5. detect scene changes and select better representative frames;
6. run OCR on selected frames if text is present;
7. compute music-presence and audio-energy heuristics;
8. send transcript, OCR, frame images, and timing data to OpenAI for multimodal analysis;
9. reconstruct the likely scenario / beat structure of the video;
10. generate embeddings for semantic retrieval;
11. save the final structured result into a database collection.

The implementation should be **robust, idempotent, retryable, and production-oriented**.

---

# Tech Constraints

## Runtime

* Node.js backend only
* TypeScript preferred
* `ffmpeg` and `ffprobe` invoked through subprocess execution
* OpenAI official Node SDK for API access

## Storage

* The final consolidated document must be stored in collection:

  * `video_analyses`

Optional auxiliary collections are allowed, but `video_analyses` must contain a complete final record.

---

# Input

The pipeline receives:

* `videoId`: internal identifier
* `filePath`: absolute or canonical local path to the downloaded video file
* optional:

  * `sourceUrl`
  * `platform`
  * existing metadata

Assume the file already exists and is readable.

---

# Output

The pipeline writes one final analysis document to `video_analyses`.

If the pipeline is rerun for the same `videoId`, it must **upsert** rather than create duplicates.

# High-Level Pipeline

Implement the following stages.

---

## 1. Probe Video

Use `ffprobe` to extract:

* duration
* width
* height
* fps
* video codec
* audio stream presence
* audio codec
* bitrate if available
* file size

Store under `mediaMetadata`.

If probing fails:

* mark pipeline as failed
* persist normalized error details

---

## 2. Extract Audio

Use `ffmpeg` to extract an audio track suitable for transcription and audio heuristics.

Preferred extraction:

* mono is acceptable
* 16 kHz or 24 kHz is acceptable
* WAV is preferred for analysis quality

Example:

```bash
ffmpeg -y -i input.mp4 -vn -acodec pcm_s16le -ar 16000 -ac 1 output.wav
```

Store:

* extracted audio path
* extraction status
* duration if available

If no audio stream exists:

* continue without transcription
* mark transcription and audio-analysis stages as `skipped`

---

## 3. Transcribe Speech with OpenAI

Use OpenAI audio transcription for the extracted audio. OpenAI’s current docs cover audio and speech workflows as part of the platform capabilities. ([OpenAI Platform][1])

Requirements:

* obtain full transcript text
* capture timestamps if available
* capture language if available
* normalize provider output into internal schema

Store under:

* `transcript.text`
* `transcript.language`
* `transcript.segments`
* `transcript.provider = "openai"`

If transcription fails:

* do not abort the full pipeline
* continue with visual analysis
* mark transcript stage as `failed`

---

## 4. Scene-Change Detection and Representative Frame Selection

Do **not** rely on naive uniform frame sampling alone.

Implement a better frame-selection strategy that combines:

* dense sampling in the first 3 seconds
* scene-change detection
* full-video coverage
* redundancy reduction

### Required behavior

### 4.1 Detect scene changes

Use `ffmpeg` scene detection or a comparable method to identify likely cuts / transitions.

Acceptable implementation approaches:

* `select='gt(scene,THRESHOLD)'`
* or another deterministic scene-change extraction strategy

For each candidate scene change, capture:

* timestamp
* frame image
* scene score if available

### 4.2 Early hook coverage

The first seconds matter disproportionately for short-form video.
Independently of scene detection, extract additional early frames:

* approximately every 0.3 to 0.5 seconds in the first 3 seconds

### 4.3 Uniform backfill

If scene detection returns too few frames, backfill using uniform sampling across the rest of the video.

### 4.4 Redundancy filtering

Reduce near-duplicate frames using one of:

* perceptual hash
* image embedding similarity
* simple histogram comparison

### Final target

Select **10 to 15 representative frames**, each with:

* `timestampSec`
* `frameIndex`
* `framePath`
* `selectionReason`:

  * `early_hook`
  * `scene_change`
  * `uniform_backfill`
  * `fallback`

Store under `frames`.

---

## 5. OCR for On-Screen Text

Run OCR on selected frames.

Expected behavior:

* OCR all selected frames, or
* first detect text-rich frames, then OCR only those

Per-frame OCR output:

* recognized text
* confidence if available
* bounding boxes if available

Also produce a merged video-level OCR summary:

* deduplicated text
* likely opening hook text
* likely CTA text
* probable subtitle/caption lines

Store under:

* `ocr.frames`
* `ocr.summaryText`
* `ocr.detected`

If OCR is unavailable:

* keep the adapter interface in place
* mark OCR as `skipped`
* do not block the rest of the pipeline

---

## 6. Audio Heuristics: Music Presence and Energy

Implement a lightweight audio analysis stage that does **not** attempt full music identification, but does estimate whether background music is present and how energetic the soundtrack is.

### Required outputs

Store under `audioHeuristics`:

* `audioPresent`
* `speechPresentLikely`
* `musicPresentLikely`
* `musicPresenceConfidence`
* `avgRmsEnergy`
* `peakRmsEnergy`
* `energyTimeline`
* `silenceRegions`
* `dynamicProfile`
* `notes`

### Suggested behavior

### 6.1 Energy extraction

Compute a time-series over the audio track:

* RMS or loudness per small window
* optionally peak amplitude
* optionally zero-crossing rate or spectral centroid if easy to add

### 6.2 Silence detection

Detect low-energy regions:

* likely pauses
* likely editing beats
* possible transitions

### 6.3 Music-presence heuristic

Implement a best-effort heuristic using audio features such as:

* continuous non-speech energy bed
* lower pause density
* repeated rhythmic energy patterns
* transcript sparsity relative to ongoing audio energy

This is a heuristic, not a classifier with guaranteed accuracy.

### 6.4 Dynamic profile

Return a coarse label such as:

* `very_calm`
* `calm`
* `moderate`
* `high_energy`
* `very_high_energy`

This stage should help the final synthesis understand pacing and soundtrack role.

---

## 7. Frame-Level Multimodal Analysis with OpenAI

Use OpenAI multimodal analysis for the representative frames. OpenAI’s current docs describe platform support for images and vision, structured output, and the Responses API. ([OpenAI Platform][1])

For each frame, obtain a structured description.

The prompt must request JSON only.

Per frame, capture:

* timestamp
* scene description
* visible people or subjects
* key objects
* visible actions
* setting / environment
* camera framing if inferable
* emotional tone
* visible text summary
* likely story role:

  * `hook`
  * `setup`
  * `development`
  * `reveal`
  * `payoff`
  * `cta`
  * `unknown`

The output must distinguish between:

* observed facts
* reasonable inferences
* uncertainty

Do not ask the model to invent continuity between frames at this stage.

Store under `frameAnalyses`.

---

## 8. Video-Level Synthesis with OpenAI

After transcript, OCR, frame analyses, scene-change information, and audio heuristics are available, call OpenAI for final structured reasoning.

Use strict JSON output.

The final synthesis must produce:

* `summary`
* `language`
* `contentCategory`
* `narrativeStructure`
* `hook`
* `setup`
* `development`
* `twistOrReveal`
* `payoff`
* `cta`
* `visualStyle`
* `editingStyle`
* `audioRole`
* `musicRole`
* `onScreenTextRole`
* `probableScript`
* `sceneBySceneReconstruction`
* `observedFacts`
* `inferredElements`
* `uncertainElements`
* `confidence`

The model must reconstruct the likely scenario of the video from:

* transcript
* OCR results
* frame descriptions
* scene-change timing
* audio-energy and music-presence heuristics
* technical metadata

It must not present guesses as facts.

The synthesis prompt must explicitly state:

* prefer grounded observations
* keep spoken text and on-screen text separate
* use audio heuristics as supporting evidence, not ground truth
* when uncertain, say so
* return valid JSON only

---

## 9. Embeddings for Semantic Retrieval

Add an embeddings stage using OpenAI embeddings. OpenAI’s current platform docs include embeddings among the API capabilities and recommend using structured platform APIs rather than ad hoc text-only prompting for these workflows. ([OpenAI Platform][1])

### Embeddings must be generated for:

### 9.1 Video-level embedding text

Construct a canonical text representation of the video from:

* summary
* narrative structure
* hook
* probable script
* OCR summary
* transcript summary
* content category
* visual style
* audio/music role

Generate and store:

* `embeddings.video`

### 9.2 Segment / scene-level embeddings

For each scene or reconstructed segment, generate an embedding from:

* scene description
* transcript slice
* OCR slice
* story role

Store under:

* `embeddings.scenes[]`

### 9.3 Retrieval metadata

Store supporting retrieval fields:

* `embeddingModel`
* `embeddingVersion`
* `embeddingTextVersion`
* `searchText`

### Retrieval use cases

This should support:

* semantic search for similar videos
* clustering by format
* retrieval of videos with similar hooks
* reuse of scenarios and structures across content libraries

---

# Non-Functional Requirements

## Idempotency

For the same `videoId`, the pipeline must:

* upsert the existing record, or
* replace the prior final record

No duplicate final records.

## Failure Tolerance

Each stage must record its own status:

* `pending`
* `completed`
* `failed`
* `skipped`

A non-core failure should not necessarily abort the entire pipeline.

Examples:

* OCR failure: continue
* transcription failure: continue
* embeddings failure: analysis may still complete as `partial`

Core failures:

* file missing
* probe failure
* frame extraction failure

## Logging

Log:

* stage start/end
* elapsed time
* subprocess commands
* OpenAI request attempts
* OpenAI validation failures
* DB writes

Do not log secrets.

## JSON Validation

All OpenAI outputs must be schema-validated before DB insertion.

If invalid JSON is returned:

* retry once with a repair prompt
* if still invalid, store raw model output in a debug field
* mark the stage as `failed`

# Detailed Service Contracts

## `probeVideo(filePath: string)`

Returns normalized media metadata.

Must validate:

* file exists
* file is non-empty
* file is a supported video container

---

## `extractAudio(filePath: string)`

Returns:

* audio path
* extraction metadata
* stage status

---

## `transcribeAudio(audioPath: string)`

Uses OpenAI transcription.

Returns normalized transcript object.

---

## `detectScenes(filePath: string, durationSec: number)`

Returns scene candidates:

* timestamp
* frame path
* scene score if available

---

## `extractRepresentativeFrames(...)`

Consumes:

* duration
* scene candidates
* early-hook sampling rules

Returns 10–15 selected frames with reasons.

---

## `dedupeFrames(frames)`

Removes near-duplicates.

Use:

* perceptual hash preferred
* fallback similarity heuristic acceptable

---

## `runOcrOnFrames(frames)`

Returns per-frame OCR and merged OCR summary.

---

## `analyzeAudio(audioPath: string, transcript?: Transcript)`

Returns:

* energy timeline
* silence regions
* music presence estimate
* dynamic profile
* notes

---

## `analyzeFramesWithOpenAI(frames, transcriptExcerpt?, ocrExcerpt?)`

Uses OpenAI multimodal reasoning through the Responses API and requests strict JSON output. OpenAI’s current docs explicitly list the Responses API, structured output, and images/vision as first-class platform concepts. ([OpenAI Platform][1])

Batch frames if needed to control payload size.

---

## `synthesizeVideoAnalysis(input)`

Consumes:

* media metadata
* transcript
* OCR summary
* frame analyses
* scene timings
* audio heuristics

Returns final structured interpretation.

---

## `generateEmbeddings(input)`

Generates:

* video-level embedding
* scene-level embeddings
* retrieval text fields

---

## `saveVideoAnalysis(document)`

Upserts by `videoId` into `video_analyses`.

# OpenAI Prompt Requirements

## A. Frame-Level Prompt

System instruction:

```text
You are a multimodal video frame analysis engine.
Analyze the provided frame and return JSON only.
Separate observed facts from inferences.
Do not invent events not visible in the frame.
If visible text exists, summarize it separately.
Classify the frame’s likely story role using one of:
hook, setup, development, reveal, payoff, cta, unknown.
```

Expected JSON:

```json
{
  "sceneDescription": "",
  "subjects": [],
  "objects": [],
  "actions": [],
  "environment": "",
  "cameraFraming": "",
  "emotionalTone": "",
  "visibleTextSummary": "",
  "storyRole": "unknown",
  "observedFacts": [],
  "inferences": [],
  "uncertainties": []
}
```

---

## B. Final Synthesis Prompt

System instruction:

```text
You are a grounded short-video analysis engine.

You are given:
1. transcript of spoken audio,
2. OCR text found on screen,
3. frame-by-frame visual descriptions with timestamps,
4. scene-change timing information,
5. audio heuristics including energy and music-presence estimates,
6. basic video metadata.

Your task is to reconstruct the likely scenario and structure of the video.

Rules:
- Return valid JSON only.
- Distinguish observed facts from inferences.
- Spoken text and on-screen text are separate channels and must not be merged unless clearly aligned.
- Audio heuristics are supporting evidence, not certainty.
- Do not claim certainty where evidence is weak.
- Prefer concise, structured, production-usable output.
```

---

## C. Embedding Text Construction Rules

Before generating embeddings, construct canonical retrieval text.

### Video-level `searchText`

Concatenate normalized sections such as:

* summary
* hook
* category
* structure
* OCR summary
* transcript summary
* probable script
* visual style
* music/audio role

### Scene-level embedding text

For each scene:

* story role
* time range
* scene description
* spoken text slice
* OCR slice

Keep this deterministic so embeddings remain comparable over time.

# Pipeline Entry Point

Implement a single orchestrator:

```ts
async function processVideo(
  videoId: string,
  filePath: string,
  sourceUrl?: string,
  platform?: string
): Promise<VideoAnalysisDocument>
```

This function should:

* run all stages
* collect normalized outputs
* upsert the final record
* return the saved DB document

---

# Acceptance Criteria

The implementation is complete when:

1. a local video file can be processed end-to-end on the Node.js backend;
2. metadata, transcript, scene candidates, representative frames, OCR output, audio heuristics, multimodal analysis, and embeddings are generated;
3. one final structured document is upserted into `video_analyses`;
4. partial failures are reflected in stage statuses rather than silently ignored;
5. the stored result is usable for:

   * semantic retrieval
   * similar-video search
   * downstream reporting
   * script generation

---

# Recommended Engineering Notes

* Prefer TypeScript.
* Prefer Zod for schema validation.
* Prefer explicit retry wrappers around OpenAI calls.
* Keep OCR abstracted behind an adapter.
* Keep frame extraction and scene detection deterministic.
* Treat music-presence output as heuristic only.
* Keep embedding generation separate from final synthesis so retrieval can be reindexed later without rerunning full video analysis.
